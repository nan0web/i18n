# @nan0web/i18n — Model-First i18n Architecture

> Цей документ є авторитетним описом архітектурного принципу **"Тексти тільки з моделей"**.
> Посилання з: [`nan.web/system.md`](../../system.md) § Універсальна Локалізація, п.4

---

## Принцип

**Всі написи UI визначаються виключно у моделях або компонентах.**

```
✅ Модель ($Component YAML) → i18n ключ → UI компонент рендерить
❌ Hardcoded строки в IDE / Sandbox / docs-site / layout
```

Ніякого hardcoded тексту у:

- IDE (Master IDE, Sovereign Workbench)
- Sandbox (OlmuiInspector)
- Scaffold / layout коді
- Docs-site генераторах

## Джерело ключів

`@nan0web/i18n` автоматично збирає ключі з:

1. **Моделей домену** — `Model.field.help` → `t(model.help)`
2. **YAML-маніфестів компонентів** — `$Component:` секція → пропи, варіанти, описи
3. **Компонентних UI-файлів** — виклики `t("...")` у коді

Це **єдині** три джерела для перекладних ключів. Все інше — ЗАБОРОНЕНО.

## $search — Пошукові теги (per-language)

Кожна мова зберігає локалізовані пошукові теги для компонентів:

```yaml
$Button:
  $search: 'кнопка натиснути дія клік submit'
  label:
    default: Click Me
  variant:
    - primary
    - secondary
```

### Правила

- Поля з префіксом `$` — **мета-поля**. Вони не потрапляють у `propTypes` / `defaultProps`
- `$search` використовується IDE/Sandbox для пошуку будь-якою мовою
- Кожна локаль має свої теги: `uk/$Component.$search` ≠ `en/$Component.$search`

### Обробка у коді

```js
// main.js — пропускаємо мета-поля
for (const [key, val] of Object.entries(config)) {
  if (key.startsWith('$')) continue // $search → skip
  // ...propTypes, defaultProps
}

// searchTags потрапляє в manifest окремо
searchTags: config.$search || ''
```

## Універсальна Архітектура Даних (Language-Agnostic Fetching)

- **Єдине Джерело (Single Origin)**: Переклади (`t`), конфігурація доступних мов (`langs`) та метадані каталогу завантажуються виключно через прямий стандартний запит (`fetch()`) до документа конкретної сторінки або компонента (наприклад, з його `_.json` копії). Це єдина точка збору загального контексту.
- **Мовонезалежність БД (Lang-Agnostic)**: Ця архітектура зберігання даних та отримання перекладів повністю незалежна від інструментів чи мов програмування. Логіка отримання перекладів ідентична як у JavaScript (Node.js/Браузер), так і в Python, Go, Rust або Swift: достатньо лише зробити банальний HTTP GET чи зчитати JSON-документ сторінки. Не потрібні жодні специфічні для конкретної мови або фреймворку i18n-плагіни.

## Каскадний Fallback

```
1. Локальний словник продукту → t.json
2. Батьківський словник → _/t.json
3. Кореневий словник → uk/_/t.json
4. Англійський ключ (як текст за замовчанням)
```

## Аудит

```bash
# Перевірка покриття перекладів
pnpm i18n:audit

# Синхронізація ключів з коду
pnpm i18n:sync
```

`extract()` знаходить `t("...")` у коді → `auditTranslations()` порівнює з наявними перекладами → `syncTranslations()` додає відсутні.
